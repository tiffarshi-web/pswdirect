## Canonical Order Paths

### Path A: Client Booking (Stripe)
1. Client submits form â†’ `create-booking` edge function â†’ booking inserted (status=pending, payment_status=invoice-pending)
2. Stripe PaymentIntent created client-side â†’ client confirms payment
3. Stripe sends `payment_intent.succeeded` â†’ `stripe-webhook` edge function
4. Webhook: updates `payment_status=paid`, saves `payment_method_id` + `customer_id`
5. Webhook: idempotency check against `dispatch_logs` â†’ calls `notify-psws`
6. `notify-psws`: geocodes â†’ `get_nearby_psws` RPC â†’ filters (gender/language/transport) â†’ push + email + SMS + in-app
7. PSW claims: `shiftStore.claimShift()` â†’ atomic `.is("psw_assigned", null)` check
8. PSW signs out: status â†’ completed â†’ trigger fires `upsert_payroll_entry_for_booking()`

### Path B: Admin-Created Order (invoice-pending)
1. Admin submits ManualOrderCreation â†’ `create-booking` (payment_status=invoice-pending, no Stripe PI)
2. `create-booking` dispatches to `notify-psws` immediately (bypasses Stripe gate)
3. PSW claims â†’ same flow as Path A step 7+

### Path C: Unserved â†’ Manual Fulfillment
1. Client booking finds no PSWs in radius â†’ `unserved_orders` row created (client-side)
2. Admin assigns PSW + generates payment link token
3. Client pays via `/pay/:token` â†’ `fulfill-unserved-order` creates booking
4. Booking created as `paid` + assigned â†’ status=active

## Idempotency Protections
- Stripe webhook: checks `dispatch_logs` for existing `booking_code` before dispatching
- Invoice email: checks `email_history` for existing `psa-client-invoice` template before sending
- Payroll trigger: uses `ON CONFLICT (shift_id) DO UPDATE` for upsert safety
- Unclaimed alerts: checks `notifications` for existing `unclaimed_alert` type before inserting
- **Unreconciled payments**: `unreconciled_payments.stripe_payment_intent_id` is UNIQUE so the same orphan PI never appears twice in the recovery queue.

## Payment Reconciliation Safety Net
Every successful Stripe `payment_intent.succeeded` MUST end in one of two states:
1. A booking row updated to `payment_status='paid'` (happy path), OR
2. A row in `unreconciled_payments` with `status='open'` (orphan path â€” admin recovery required).

The `stripe-webhook` function enforces this. If the PI metadata is missing `booking_id`/`booking_code`, OR the lookup fails (no matching booking), the webhook records the PI + customer details in `unreconciled_payments` so admins can resolve via **Orders & Operations â†’ Recovery Queue**. Resolution links the orphan to the correct booking via the `admin_resolve_unreconciled_payment` RPC; dismissal uses `admin_dismiss_unreconciled_payment` (status `refunded` or `ignored`). The webhook also handles `payment_intent.payment_failed` for observability.
